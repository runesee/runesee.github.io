Kylling&Ris

Kylling&Ris (Chicken&Rice) is a website Ã  la myfitnesspal, which lets users search for and register food they have eaten. The website allows searching, filtering and sorting a set of data gathered from a REST-API containing Norwegian groceries: Kassal.app. Users kan easily register food elements by clicking the related plus button. Afterwards the users can include the weight of the product and select a date. The product will then be shown in a table, which allows easy navigation between dates through a calender. In the table users can remove or edit registered elements using the recycle bin and pencil icons, respectively.

During this project I mainly worked on the frontend. One of the requirements for this project was to have a responsive design. While we did not follow a mobile-first approach, we did a lot of testing to ensure the application worked well on various screen sizes. I personally spent a lot of time ensuring the three pages all had dynamic, sleek designs. This was a bit tricky as the dynamic loading of data would add extra elements to the page, and we used a third party component for the dynamic scrollable list.

TL;DR

An interactive calorie counter app using Neo4J and GraphQL to store and present 7500 Norwegian groceries gathered from an API. Users can log food eaten on specific dates, and the data is saved in the database. The design is responsive and works on both mobile and larger screens. Made using React, Vite and Typescript.

Technical requirements

  • The Application is made with React, Typescript and set up with Vite.
  • State management using redux. Search settings and the log date are saved.
  • Uses imported third party components like datepicker-calender and infinite scroll elements.
  • Uses Neo4j for backend database. Uses GraphQL, Apollo Server and Apollo Client for queries. Approx. 7500 different food objects stored in the database.
  • Prettier has (mostly) been used during the project to format code. Prettier and Eslint were run on the entire project before the final hand-in.
  • We set up tests for the frontend, and backend is tested (somewhat) in end-to-end test of food search. We also manually tested queries and mutations with Apollo Server sandbox.
  • Comments were used during the project to explain the code.

Usability & Availability

  • Added aria-labels to support screen readers.
  • Suitable text distance and choice of colours with clear contrast to ensure readability.
  • Added text alternatives to non-text content. All essential functionality is also accessible by keyboard.
  • Navigation is simple as the website only consists of three pages.
  • We have tried to limit overuse of the div/span tag.
  • Added "Kaloriteller" (Calorie counter) as a sub text in the logo to ensure that users understand the purpose of the application.
  • Added descriptive help text in the food log when the user has not added any items.
  • Added different styling when components are focused to communicate to the user that the components can be interacted with.
  • The website helps the user avoid errors. It is for instance not possible to add food in the future.

Functional requirements

  • Possible to search using a search bar.
  • List based presentation of search with dynamic loading of additional results through scrolling.
  • Relevant info about food object is shown by searching or in the log table.
  • Each food object in search results can be clicked to see more info.
  • Results can be sorted and filtered (allergies, protein and such).
  • The user generated data consist of the food log and its related user.
  • Data is saved on the server. If you do not make a user, you are given a guest user so that the data is still saved on the server.
  • The application has a responsive design.
  • The database and backend was originally set up on a virtual machine provided by the course staff. As we no longer have access to it, the backend part of the application will not work.

This image shows the mobile version of the main page. The search and log elements are divided into each their own tabs. We originally wished to have the tab buttons be on the bottom of the view, as it would be easier to reach with your thumbs on mobile. I unfortunately eventually had to give up on this due to limitations with the third party component we used.

In the top right you can see we are currently "logged in" as a guest user. Data from guests are also stored in the database. We discussed only saving this data in local storage, but figured the examinators might not bother creating an account to test our app. Therefore it would be better to store all user generated data in the database, even if not logged in.

The other pages of the application are the 'register user' and 'log in' pages. The former can be seen here. These pages can be accessed by clicking the account icon on the main page.

Security was not prioritized in this project. To us, the app's usability was a much greater concern. Further, the application was only intended to be running during this project, and the course did not specify any security requirements. Passwords were therefore stored in cleartext in the database.

During this project I learned a lot about responsive frontend design. It was at times very tedious, and it took a bazillion errors before I got it finally right. In the end, we were very happy with the design. We decided to use Typescript for this project, as we had never used it before. It was not very different from JS, and easy to use with React.

One thing I regret is only working on the frontend. While all group members would use the Apollo client to view the database content and try queries, one person more or less single handedly set up our entire backend. We greatly appreciated this at the time, but in hindsight it would have been nice to gain more backend development experience.
